Python GIL(Gloabal Interpreter Lock)

为了支持多线程机制，一个基本的需求就是需要实现不同线程对共享资源访问的互斥。这正是Python引用GIL的根源。GIL是一个解释器，在一个线程拥有了解释器的访问权后，其他的所有线程都必须等待它释放解释器的访问权。Python的多线程是在GIL的基础上实现的。意味着无论如何，在同一时间，只有一个线程可以访问Python所提供的API，在单核处理器上，GIL的保护粒度很大，若是保护粒度过小，就会出现频繁的加锁、解锁操作，但是加锁解锁对于操作系统来说是比较重量级的操作，性能反而不如保护粒度大的GIL。另一方面，GIL的存在限制了多核处理器，导致多核处理器退化为单核处理器。所以Python并不能发挥多核处理器的作用。

对于Python而言，字节码解释器是Python的核心所在，所以Python通过GIL来互斥不同线程对解释器的使用。当某个线程（A）获得GIL之后，其他线程必须等待A释放GIL后，才能进入解释器。

对于线程调度机制，最关键的是要解决两个问题：

1、在何时挂起当前线程，选择处于等待状态的下一个线程

2、在众多处于等待状态的候选线程中，选择激活哪一个线程

对于问题1，Python内部的时钟N规定了Python在执行了N条指令以后应该立即启动线程调度机制。在设置中，这个值为100.

当一个线程获得了访问Python解释器所必须的GIL并进入解释器后，Python内部的监测机制就开始启动，当这个线程执行了100条指令后，Python解释器将强行挂起当前线程，开始切换到下一个处于等待状态的进程

对于问题2，选择激活哪一个线程Python并没有做干涉，而是借用了底层操作系统所提供的线程调度机制来决定下一个进入Python解释器的线程。也就是意味着Python的线程就是操作系统所支持的原生线程，对于不同的操作系统，有着不同的实现。但是Python提供了一套统一的抽象机制，这就是Python中的 thread 以及threading。

创建线程

初始化环境

Python在启动时，并不支持多线程，只有在调用到 _thread.start_new_thread时（Python3中为 _thread），会明确指示Python虚拟机创建新线程，Python虚拟机就会自动建立多线程机制所需要的数据结构、环境和GIL。

GIL的真面目为void*。创建工作由PyThread_alloc_lock完成，线程通过PyThread_acquire_lock尝试获取GIL。

PyThread_acquire_lock有两种工作方式，通过参数 waitflag 来区分。waitflag指示当GIL当前不可获得时，是否进行等待，更直接的说，就是当前线程是否通过WaitForSingleObject将自身挂起，直到别的线程释放GIL，然后由操作系统将自己唤醒

如果 waitflag 为0，Python就会检查当前GIL是否可用。其中指示GIL是否可用的变量为owned，owned初始化为 -1，意味着GIL可用。当线程要使用GIL时，必须将其设置为0，当owned设置为0后，代表当前GIL已经被一个线程所占用。当有一个线程在等待GIL时，owned的值加1。当一个线程释放GIL时，一定会将owned的值减1。当所有线程都释放GIL后，owned会再次变为-1，意味着GIL再次变为可用。

创建线程

主线程使用PyThread_start_new_thread来创建子线程。主线程通过调用操作系统来成功创建出原生线程，当成功创建并返回后，主线程开始将自己挂起，开始进入子线程（在这里子进程还没有开始申请GIL），执行bootstrap函数，在bootstrap中，子进程完成三个动作，获取线程ID，通知obj.done，调用t_bootstrap，其中，子线程通过obj.done唤醒主线程。为什么Python需要让主线程等待子线程的通知呢？是因为PyThread_start_new_thread需要返回子线程的id，子线程只有子线程被激活后才可以在子线程中获取。所以主线程等待的就是子线程的ID，当子线程中设置了obj.id，就会设法唤醒主线程。

Python通过PyThread_start_new_thread来完成创建子线程的操作。在通过操作系统创建出子进程后，主线程会将自己挂起，进入子进程（这时子进程并没有获得GIL），等待子进程中控制的obj.done。子线程的过程是bootstrap，在bootstrap中，子进程完成了三个动作：获取线程ID、通知obj.done、调用t_bootstrap。那么Python为何要让主线程等待子线程的通知呢？这时因为PyThread_start_new_thread需要返回所创建子线程的ID，而这个ID只有在子线程被激活后才能在子线程中获取。因此主线程等待的就是这个ID，一旦子线程设置了obj.id，就会设法唤醒主线程。

当主线程返回子线程ID后，就会继续执行后面的字节码，这时，主线程拥有GIL，子线程则进入t_bootstrap，并最终进入等待GIL的状态。开始了与主线程对GIL的竞争。在t_bootstrap中，第一个进行的动作就是PyEval_AcquireThread来申请GIL，当PyEval_AcquireThread结束后，子线程就获得了GIL，开始执行后面的内容。

线程状态保护机制

这里的线程状态就等同于进程的上下文，在进行线程切换时，每一个线程都需要获得自己线程的相关信息。Python内部，维护着一个全局变量：PyThreadState* _PyThread_State_Current。当前活动的线程所对应的线程状态对象就保存在这个变量里。当Python进行线程调度时，会被激活的线程状态就会被保存在这个变量里。

Python在进行线程调度时，如何获得被激活线程的状态

Python内部会有一个单项链表来管理所有的Python线程状态对象。需要寻找一个线程对应的状态对象时，就遍历这个链表。对于状态对象链表的访问，不必在GIL的保护下进行。因为Python会创建一个独立的锁，专门对状态链表进行保护。这个锁的创建是在Python进行初始化的时候创建的。链表中Node的数据结构为：

id存放着线程的id，value指向PyThreadState。

子线程在创建了自身的线程状态对象后，会通过_PyGILState_NoteThreadState（该函数在PyThreadState_New中）将这个对象放入状态对象链表中。但是当前线程不一定获得了GIL，子线程到现在还没有申请GIL，自然不会将自己挂起，所以操作系统可能在主进程和子进程之间进行切换。但是，Python级的线程调度和操作系统级别的线程调度是不同的，Python级别的线程调度意味着GIL所有权的易手，但操作系统并不是。当所有的线程都完成了初始化动作后，操作系统的线程调度才会和Python的线程调度一致。那时，Python的线程调度会迫使当前进程释放GIL，这一操作会触发GIL中维护的Event(Windows)内核对象，这个触发又进而触发操作系统的线程调度。

完成PyThreadState_New后，就到了PyEval_AcquireThread，到了这一步，子线程开始争取GIL，到了这一步，子线程将自己挂起，操作系统的线程调度机制不能靠自己将其唤醒，只有等待Python的线程调度机制强迫主线程放弃GIL后，子线程才会被唤醒。

当子线程被Python线程调度机制唤醒后，第一件事就是通过PyThreadState_Swap将Python维护的当前线程状态设置为自身的状态对象。然后进入PyEval_CallObjectWithKeywords，从这里一直往前，最终调用PyEval_EvalFrameEx，进入解释器。

线程调度

标准调度

Python的线程调度机制是内建在Python解释器核心PyEval_EvalFrameEx中的。

其中_Py_Ticker就是Python内部通过软件模拟实现时钟中断机制，而_Py_CheckInterval保存着Python设定的指令数，也就是通过sys.getcheckinterval()得到的值。PyEvalFrameEx每执行一条字节码指令，_Py_Ticker就减1，当执行了_Py_CheckInterval条指令后，_Py_Ticker减少到0，进入线程调度。

当_Py_Ticker减少到0时，主线程先将维护当前线程状态对象的_PyThreadState_Current设置为NULL，然后释放掉GIL。这时，由于等待GIL而被挂起的子线程被操作系统的线程调度机制唤醒。从而最终进入PyEval_EvalFrameEx，对于主线程，虽然它已经失去GIL，但是由于他没有被挂起，所有对于操作系统的调度机制，它还可以被再次切换为活动线程。

当操作系统的调度机制将主线程切换为活动线程后，主线程将再次执行PyThread_acquire_lock，主线程申请GIL，由于被子线程占有，主线程将自身挂起，从这时开始，操作系统的线程调度不会再将主线程切换为活动线程，除非等到子线程释放GIL之后。

阻塞调度

另一种调度方式为阻塞调度，当线程A通过某种操作，比如说等待输入，将自身阻塞后，Python应该将等待GIL的线程唤醒。

Python子线程的销毁

在线程完成全部的计算后，Python将销毁线程。Python主线程和子线程的销毁是不同的，因为主线程的销毁必须要销毁Python的运行时环境，而子线程测不需要进行这些动作。

Python首先会通过PyThreadState_Clear清理当前线程所对应的线程状态对象，其实就是对线程状态对象中维护的东西进行计数引用的维护。随后Python释放GIL，释放GIL的操作是在PyThreadState_DeleteCurrent中完成的。PyThreadState_DeleteCurrent首先会删除当前线程状态对象，然后通过PyEval_ReleaseLock释放GIL，最后的PyThread_exit_thread是一个和平台相关的操作，完成各个平台上不同的销毁原生线程的工作。
