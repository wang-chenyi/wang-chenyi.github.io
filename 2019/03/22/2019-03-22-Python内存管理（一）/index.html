  1. 操作系统的内存管理接口

  2. Python在底层的管理接口上所做的包装(为了平台可移植性，保证相同的语义由相同的运行时行为，为了处理于平台相关的内存分配行为时，要在C的内存分配接口上再提供一层包装)

小块空间内存池

在Python中，整个小块内存的内存池可视为层次结构，从下至上分别为 block、pool、arena、内存池。

pool中申请内存

pool_header中定义的 freeblock 指向的是下一个可用的 block 的起始地址，每申请分配一个block，返回当前的 freeblock，就是该分配的 block 起始地址，然后 freeblock 和 nextoffset 都会向后移动一个 block 的距离，申请，前进，申请，前进。而 maxnextoffset 指定了最后一个block的边界。当nextoffset > maxnextoffset时，说明一个pool中的block已经被遍历完了。

在释放一个block之前，freeblock指向的是第一个未分配的block，

假如我们申请了五个block内存块，freeblock指向的就是第六个block，虽然freeblock指向一个有效的block内存块，但是 freeblock其实是为空的（也就是第六个block为空）。如果过了一段时间，程序释放了第二个和第四个，这时，freeblock会将第二块和第四块内存加入freeblock链表中，该链表称为离散自由block链表，当程序又要继续分配内存时，就会先分配已经释放的第二块block，再分配第四块内存，直到 freeblock == NULL，表明到了链表尾部。开始原本的分配方式。

arena

arena中arena_object的定义(类似于pool_header)

在Python中，确实会存在多个arena_object构成的集合，但是这个集合构成的是数组，数组首地址由arenas维护。在arena_object中存在nextarena和prevarena，这两个变量有其他的作用，

虽然arena是用来管理多个pool的，但是他和pool管理block的方式不太一样，因为一个pool代表4kb内存，当一个pool被申请后，它包含的block也一定被申请了。但是arena_object和它管理的内存是分离的，当arena_object被申请时，它所管理的pool集合的内存并没有被申请，而是在后面某一时刻，arena_object于pool建立连接。建立连接的过程被Python一刀切下分为两种状态。

  1. 未使用状态（unused）：arena_object没有与pool建立连接
  2. 可用状态（usable）：arena_object与pool建立了连接，转换为usable状态

未使用的arena的链表表头是unused_arena_objects，arena和arena通过nextarena连接，是一个单向链表。

可用的arena链表表头是usabel_arenas，arena和arena之间通过nextarena和prevarena连接，是一个双向链表。

arena操作

初始化arenas数组，初始化长度为16，后每次申请翻倍。初始化后的所有arena都在unused_arena_objects单链表里面。

初始化arena后，变可以从中取出一个pool，如果是全新的arena，则会取出第一个pool，如果不是全新的arena，则会判断arena->freepools中是否有之前释放掉的pool，因为一个pool释放时，会加入到所在arena->freepools链表的最前方。

每分配一个pool，就检查是不是用到了最后一个，如果是，需要变更usable_arenas到下一个可用的节点，如果没有可用的，那么下次进行内存分配的时候, 会判定从arenas数组中取一个

回收arena

  1. arena中所有pool都是闲置的(empty), 将arena内存释放, 返回给操作系统
  2. 如果arena中之前所有的pool都是占用的(used), 现在释放了一个pool(empty), 需要将 arena加入到usable_arenas, 会加入链表表头
  3. 如果arena中empty的pool个数n, 则从useable_arenas开始寻找可以插入的位置. 将arena插入. (useable_arenas是一个有序链表, 按empty pool的个数, 保证empty pool数量越多, 被使用的几率越小, 最终被整体释放的机会越大)
  4. 其他情况, 不对arena 进行处理

内存池

虽然arena是Python内存池的最上层结构，用来管理pool，但是申请内存的基本操作单元依然是pool，比如在我们申请28字节的内存时，Python会在内存池内寻找一块能够满足需求的pool，从中取出一个block返回，而不会去寻找arena。因为pool是一个由size概念的内存管理抽象体。而arena是没有size概念的内存管理抽象体。这就意味着同一个arena，在不同的时刻，管理的pool可能都是32字节的，也有可能一半是32字节的，一半是64字节的。这就决定了在进行分配内存和销毁时，所有动作都是在pool上完成的。

内存池中的pool，在Python运行时，总是处于三种状态中的一种

  1. used：至少有一个被使用，至少有一个未被使用。这种状态的pool受控于Python内部维护的usedpools数组
  2. full：pool中所有的block都已经被使用，这种状态的pool在arena中，但是不再arena的freepools中。
  3. empty：pool中所有的block都未被数用，处于这种状态的pool通过pool_header中的nextpool指针构成一个链表，这个链表的表头就是arena_object中的freepools。

其中的usedpools[0]， usedpool[1]表示的pool管理着尺寸为8个字节的block，对应的size class index为 0。所以usedpools[2]代表管理16字节的pool，以此类推，直到usedpools[63]。规律就是 index * 2 <--> 管理size class index为i的pool

所以，该pool在usedpools中的位置为usedpools[3 + 3]也就是usedpool[6]，但是usedpools[6]指向的却是usedpools[4]，但是usedpools[6]->next呢？就是该pool的首地址（usedpools[4]）往后偏移8个字节，就是usedpools[6]。

说来说去，也就是该pool的nextpool和prevpool都指向了自己！这样做有什么用？

如果我们要将管理32字节的pool放入usedpools中，只需要进行usedpools[i + i]->nextpool = pool即可。下次需要访问size class为3（32字节）的pool时，简单的访问usedpool[3+3]即可。Python正是使用这个usedpools快速的从众多的pool中快速寻找到一个最合适的pool来操作。

这样的好处非常明显，如果使用链表，则增加了遍历时间，使用数组可以一步到位，快速定位。其次很好的使用了nextpool的地址为该pool的首地址+8字节这个特性，不需要修改数据结构。

Python内存分配步骤(重点)

前面都是基础，现在到了正片，开始真正的内存分配。

当Python启动后，usedpools中并没有任何内容，Python采用延迟分配，当开始申请小块内存时，Python才会建立内存池。（根据中文来解读整个分配步骤）

释放block

释放block会引起pool的变化，由三种情况

  1. 释放block，pool从used状态转为used状态
  2. 释放block，pool中used状态转换为empty状态
  3. 释放block，pool中full状态转换为used状态

对于情况1：

将该block放入自由block链表中，并调整计数引用

对于情况3：

将该pool放入usedpools中即可

对于情况2：

将pool放入arena维护的freepools链表中，且如果arena中所有的pool都是empty，则释放pool集合所占有的内存，并且调整usable_arenas和unused_arena_objects链表。
